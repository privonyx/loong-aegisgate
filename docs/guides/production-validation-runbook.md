# Production Validation Runbook (Phase 5.2)

[中文版](./production-validation-runbook_zh.md)

A repeatable, copy-paste command list that takes AegisGate from source to a verified
production-profile deployment (Redis + PostgreSQL + OpenTelemetry + Guard + Control Plane),
producing deterministic evidence that every backend is truly active — not silently degraded
to memory/sqlite. Acceptance cases: [`acceptance-test-plan.md` §4.5 PROD-*](../acceptance/acceptance-test-plan.md).

## Overview

The validation has two layers (creative C1, plan E4):
- **Layer 1 (per-PR, blocking):** native build + Redis/PG service containers + `smoke-prod.sh --mode ci`, wired in `.github/workflows/ci-prod-smoke.yml`.
- **Layer 2 (nightly / on-demand, non-blocking):** full Docker Compose stack with the real Guard model + OTel export side-evidence (see last section).

This runbook is the manual / real-machine path (Layer 2-equivalent depth) you run once per release.

## Prerequisites

- A Linux host with build toolchain and `vcpkg` available (see `scripts/build.sh -t Release`).
- Network access for vcpkg dependencies and (optionally) the Guard model download.
- Ability to run Redis, PostgreSQL, and an OTel Collector locally (or reachable).
- An upstream LLM provider key (for the `--with-upstream` full-chain check).

## Step 1 — Build (production profile)

```bash
# Install build tools needed by vcpkg's libpq source build. build.sh checks
# these up front and exits before a long vcpkg run if they are missing.
sudo apt-get update
sudo apt-get install -y autoconf automake libtool autoconf-archive bison flex

# Builds the all-capabilities binary (Redis + PG + OTel + Guard + Control Plane).
# Tee the output so smoke-prod.sh can assert the capability summary.
bash scripts/build.sh -t Release 2>&1 | tee build/build.log
```

Confirm the capability summary shows five `[ON ]` rows for `ENABLE_REDIS`, `ENABLE_PG`,
`ENABLE_OPENTELEMETRY`, `ENABLE_CONTROL_PLANE`, `ENABLE_GUARD_MODEL`.

## Step 2 — Provision backends

```bash
# Installs/starts Redis, PostgreSQL and the OTel Collector with hardened defaults
# (random PG password, credentials file chmod 600).
sudo bash scripts/setup-prod-deps.sh
```

Download the Guard model if not present (required for PROD-05 `active`):

```bash
bash scripts/download_guard_model.sh models/guard
```

## Step 3 — Configure

Export secrets (never commit them) and use the production config template:

```bash
export AEGISGATE_API_KEY="your-gateway-key"
export AEGISGATE_ADMIN_JWT_SECRET="$(openssl rand -hex 32)"
export POSTGRES_URL="postgres://aegisgate:<password>@127.0.0.1:5432/aegisgate"
export REDIS_PASSWORD="<redis-password-if-any>"   # omit if Redis has no auth
# config/aegisgate.prod.yaml already requests redis + postgres + otel + guard.
# Or source the credentials generated by setup-prod-deps.sh (it uses export,
# so child processes inherit them):
#   source ~/.aegisgate_prod_creds   # injects POSTGRES_URL / PG_PASSWORD
```

> ⚠️ `POSTGRES_URL` and friends MUST be `export`ed — the gateway reads them via
> `getenv`. A plain `VAR=...` shell-local variable is invisible to the child
> process, resolves to empty → PG connection fails → strict mode refuses to start.

> Keep `storage.strict_backends: true` (the default). Do NOT set it to `false` for validation —
> you want misconfiguration to fail loudly.

## Step 4 — Start

```bash
# From source tree:
./build/src/aegisgate config/aegisgate.prod.yaml
# Or from a distribution package:
#   ./start.sh --profile prod        (or: AEGISGATE_PROFILE=prod ./start.sh)
```

Watch the startup log for `Cache store: redis`, `Persistent store: postgres`,
`OpenTelemetry tracing initialized`, and `GuardClassifier: local ONNX guard model active`.

## Step 5 — Run the smoke evidence pack

```bash
bash scripts/smoke-prod.sh --mode real --with-upstream \
  --config config/aegisgate.prod.yaml \
  --binary build/src/aegisgate \
  --cmake-log build/build.log \
  --host 127.0.0.1 --port 8080
```

Exit code `0` = all assertions PASS. All output and the `--out` evidence files are redacted
and `chmod 600`; the default `.smoke-out/` directory is git-ignored.

## Step 6 — Side evidence (real mode)

```bash
redis-cli KEYS '*' | head          # keys written
psql "$POSTGRES_URL" -c '\dt'      # tables exist
psql "$POSTGRES_URL" -c 'SELECT count(*) FROM cost_records;'   # > 0 after traffic
# OTel: check your Collector/Tempo for aegisgate traces (bump sample_ratio to 1.0 if needed).
```

## Fail-closed behavior (G1)

With `storage.strict_backends: true` (default), if the YAML requests `redis`/`postgres` but the
binary was not built with that backend, or the backend is unreachable at startup, the process
**refuses to start** (critical log + non-zero exit) instead of silently falling back to memory.
To verify PROD-08, stop Redis and start again: startup must abort with a clear message. Set
`storage.strict_backends: false` only when you explicitly want degrade-but-stay-up.

## Recording results

Copy [`prod-validation-result-template.md`](../acceptance/prod-validation-result-template.md),
fill the PROD-01..PROD-12 matrix with redacted evidence, and attach the `smoke-prod.sh` summary.

## Layer 2 — nightly full-stack (non-blocking)

For the heavier checks (real Guard model active + real OTel export to Tempo), run the Docker
Compose stack on a schedule or `workflow_dispatch`:

```bash
docker compose up -d redis postgres tempo aegisgate
docker compose logs aegisgate | grep -E "Cache store|Persistent store|guard model active|tracing initialized"
```

These do not block PRs; failures are advisory and investigated next business day.
